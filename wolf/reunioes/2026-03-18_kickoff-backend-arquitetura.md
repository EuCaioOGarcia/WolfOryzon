Autenticação e Login de Assessores Foi acordado que a plataforma utilizará a autenticação via conta Google em vez de usuário e senha, exigindo que a equipe pesquise os requisitos para implementar essa funcionalidade. O login do assessor deve integrar a conta Google com o controle de acesso e ordens na plataforma. Telas Iniciais e Fluxo do Assessor Para o MVP, foram priorizadas as telas de clientes, Home, listagem de ofertas e a tela de envio de push, que substituirá o nome "envio de ordem". A plataforma enviará o push, e o aceite acionará o backend para a transação. Detalhes Objetivo da Reunião e Alinhamento Inicial : A reunião foi marcada para definir os requisitos essenciais de cada fluxo (administrador e assessor), servindo como base para a construção das telas do sistema. O Murilo Viviani informou que tentaria ajudar com o máximo de detalhes possível, mas mencionou a possibilidade de adiar a reunião ou realizá-la novamente no dia seguinte para garantir a participação do Dudu, que estava mais familiarizado com a construção do sistema. Requisitos de Cadastro de Assessores : O primeiro tópico foi o cadastro de assessores externos pelo administrador, e a equipe discutiu quais campos seriam obrigatórios. Eles definiram que os campos básicos necessários seriam nome, e-mail e CPF do assessor, e sugeriram que o número de telefone fosse opcional. Vinculação de Empresa e Tipos de Usuário para Assessores : Foi sugerido vincular o assessor a uma empresa, o que permitiria o registro do CNPJ e da razão social para um registro total das compras. Além disso, sugeriu-se a atribuição de um "tipo" ao usuário (assessor, gerente, ou outros), o que poderia ser expandido para diferentes tipos de executores ou se o sistema for trabalhar apenas com assessores. Estrutura de Cadastro de Usuários (Genérica) : Houve uma discussão sobre como estruturar o cadastro, decidindo que seria melhor ter um "cadastro de usuário" genérico, onde o "tipo de usuário" (assessor, administrador, gerente) definiria suas permissões e acesso. Para o usuário, os dados seriam nome, e-mail, CPF, e telefone, com o tipo de usuário definindo as *roles* de acesso às telas e funcionalidades. Requisitos de Cadastro de Clientes : O cadastro de clientes deve ser feito pelo assessor e deve incluir nome, CPF, e-mail e telefone, assim como o *consumer ID*, que é essencial, pois o cliente deve ter uma conta na plataforma. O cliente não usará o sistema diretamente, apenas receberá notificações e aceitará o *push*. Autenticação de Login : Para o login, eles concordaram em usar um método de autenticação que não fosse usuário e senha, preferindo a autenticação com conta Google, o que exige que a equipe pesquise os requisitos para implementar essa funcionalidade. O assessor usará seu próprio e-mail para se autenticar, e a plataforma controlará a interface entre a conta e as ordens enviadas. Telas Iniciais para o Assessor (MVP) : O fluxo do assessor deve incluir a *Home* e uma tela de listagem de clientes como funcionalidades iniciais. Para o Produto Mínimo Viável (MVP), eles priorizaram as telas de clientes, *Home*, listagem de ofertas, e a tela de envio de ordem (ou *push*). A *Home* deve ser planejada com módulos que podem ser adicionados ou removidos futuramente. Funcionalidade e Detalhes da Ordem de Compra/Envio de Push : Foi esclarecido que a plataforma não efetua a compra em nome do cliente, mas sim envia um *push* que, se aceito, aciona o backend existente para executar a transação, devendo-se mudar o nome de "envio de ordem" para "tela de envio de push". A tela de envio de *push* deve permitir que o assessor selecione a oferta e o valor, sendo que a oferta pode ser preenchida se o acesso vier da listagem de ofertas. Detalhes do Cliente e Próximos Passos : As informações detalhadas do cliente (posição, produtos em carteira, saldo, etc.) são consideradas *core* e devem ser pensadas, enquanto sugestões de investimento (como recomendar ofertas com base no perfil) são consideradas uma evolução futura.
