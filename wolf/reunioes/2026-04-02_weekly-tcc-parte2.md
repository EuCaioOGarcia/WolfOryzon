Detalhes Visualização de Clientes e Filtragem de Dados : Giovanne Antony Bahia Torquato discutiu a necessidade de uma visualização detalhada dos clientes, incluindo seus dados pessoais e a possibilidade de aplicar filtros. Foi observado que a tela atual, que listava a carteira de investimentos, tinha sido copiada e colada do MZP, com a remoção dos gráficos. A tela exibe o rendimento consolidado (líquido e bruto), o total investido e detalha a carteira e os produtos do cliente, com a sugestão de aplicar filtros por produto ( 00:00:00 ). Ajuste de Terminologia de Ofertas para Custódia : Houve um debate sobre a terminologia "ofertas" na tela, pois as listadas já eram transações efetivadas. Foi sugerido substituir o termo "oferta" por "custódia", já que a custódia representa o produto adquirido, enquanto a oferta é o processo que precede a aquisição. Além disso, sugeriram incluir um botão em cada linha para possibilitar o reinvestimento, embora a tela atual seja apenas uma visão de administrador (ADM) ( 00:01:16 ). Fluxo de Criação de Ordem e Visão de Assessor : A tela atual, embora copiada de um fluxo de ADM, não deveria ter a funcionalidade de "criar nova ordem" nessa visão. Giovanne Antony Bahia Torquato esclareceu que a criação de ordens seria uma funcionalidade para o assessor, permitindo-lhes sugerir investimentos, como um pós-fixado, caso o cliente tenha potencial. O fluxo de criação de clientes seria delegado ao próprio assessor, que depois teria um local para aprovar edições ( 00:02:09 ). Funcionalidade de Cadastro de Assessores e Código Único : A administração de assessores inclui uma seção na barra lateral para visualizar e cadastrar assessores. O cadastro completo incluiria e-mail, CPF, telefone e empresa, além da possibilidade de importação via CSV ou Excel. Foi considerada a necessidade de um código único para o assessor, como um "consumer ID", para facilitar a associação e pesquisa ( 00:03:05 ). Preenchimento de Identificadores Únicos e Priorização de Estrutura : Foi discutido que o código único do assessor poderia começar com uma letra (como 'A') seguida por um número (ex: A1234) e poderia ser gerado automaticamente, evitando a criação manual ( 00:04:09 ). Giovanne Antony Bahia Torquato enfatizou que, embora a identificação seja importante, o foco inicial deve ser na estrutura, pois a interface é o mais simples de mudar ( 00:05:16 ). Funcionalidades Mínimas da Home e Listagem de Clientes para Assessores : A página inicial (home) para o assessor ainda não estava definida, mas uma funcionalidade essencial seria a listagem de clientes. Essa lista deve incluir o status do cliente, indicando se eles iniciaram ou não o processo de assessoria ( 00:05:57 ). Um cliente estaria "vinculado" se estivesse na lista, mas o status "iniciado" indicaria que o assessor já está gerenciando ativamente a carteira ( 00:06:44 ). Debate sobre Acesso de Dados do ADM e Privacidade : Houve um questionamento sobre o nível de acesso do administrador (ADM) aos dados detalhados dos clientes. O consenso foi de que o ADM não deveria ter acesso a nada além da listagem de assessores e, possivelmente, uma listagem dos clientes de cada assessor para fins de gerenciamento e auditoria. A preocupação principal é com a privacidade e o risco de vazamento de informações ( 00:07:31 ). Estratégia de Desenvolvimento para Limitar o Escopo de Dados do ADM : Foi decidido seguir o princípio da "navalha de Occam", focando na solução mais simples e estritamente necessária ( 00:09:22 ). A tela de dados pessoais detalhados do cliente para o ADM não é considerada necessária. O foco da tela do ADM seria o gerenciamento básico dos assessores (ativar/inativar) e as telas de cadastro e importação ( 00:10:12 ). Princípio de Desenvolvimento Enxuto e Versão Mínima Viável (MVP) : Giovanne Antony Bahia Torquato aconselhou a equipe a evitar o excesso de funcionalidades e a focar no essencial, seguindo o princípio de que "o bom é inimigo do ótimo" para a entrega do MVP ( 00:11:46 ). Foi sugerido remover a funcionalidade de notificações para o administrador por não ser essencial ( 00:13:17 ). Estrutura da Home do Assessor e Visão da Carteira : Para o assessor, a home poderia ser a própria página de listagem de clientes, com opções de busca e filtro por status ( 00:14:03 ). Ao clicar no cliente, a tela de carteira exibiria informações essenciais, focando no mínimo para o MVP, excluindo gráficos. Nessa tela, as "ofertas" devem ser referidas como "custódias" ( 00:14:43 ). Fluxo de Criação de Ordem e Vitrine de Ofertas : A criação de uma nova ordem requer a implementação de um fluxo separado, como um *popup* ( 00:14:43 ). A equipe também considerou a necessidade de uma "vitrine de ofertas" mostrando o que está disponível, com a ressalva de que o conteúdo, como a identificação da empresa (PicPay/JF Investimentos), seria ajustado ( 00:15:37 ). Validação do Fluxo de Ofertas com o Consumer ID : O fluxo de listagem de ofertas deve estar associado a um cliente específico, pois a API de ofertas exige o `Consumer ID` como parâmetro obrigatório ( 00:17:24 ) ( 00:25:56 ). Sugeriu-se que a vitrine de ofertas, se implementada, fosse uma aba dentro da visão do cliente, trazendo a listagem com base no `Consumer ID` ( 00:27:32 ). Oferta Combo e Oportunidades para Assessores : Foi discutida a funcionalidade de "Oferta Combo", onde a compra de uma oferta pode habilitar um conjunto adicional de ofertas exclusivas ( 00:30:41 ). Giovanne Antony Bahia Torquato indicou que isso é crucial, pois pode ser usado para criar ofertas combo exclusivas para assessores, mas a implementação deve ser cautelosa para não listar ofertas sem o `Consumer ID` ( 00:31:56 ). Sugestões para a Tela Home do Assessor (Visão Geral) : A tela *home* poderia incluir um resumo executivo com métricas como patrimônio sob gestão, clientes ativos e receita estimada, além de alertas de vencimentos próximos de títulos ( 00:33:45 ). O foco em ações necessárias, como clientes com títulos vencendo, geraria a necessidade de telas específicas para que o assessor possa interagir com esses clientes ( 00:34:32 ) ( 00:37:05 ). Desenvolvimento e Limitação do Escopo da Home : Giovanne Antony Bahia Torquato sugeriu que a tela *home* não seja o ponto de partida do desenvolvimento, pois não seria possível popular os dados sem clientes ( 00:37:53 ). A criação de metas e alertas na *home* (como captação líquida) também deve ser tratada com cautela, pois pode gerar novas exigências, como uma tela de criação e aprovação de metas ( 00:38:35 ). Modelagem de Banco de Dados: Estrutura Inicial e Usuários : A equipe apresentou a modelagem inicial do banco de dados, incluindo a tabela `User` (para ADM e Assessor) e `Client` ( 00:40:39 ). Foi proposto o uso de uma tabela intermediária (`Advisor/Client` ou similar) com um `Status` (ativo/inativo) para preservar o histórico e auditar todas as movimentações ( 00:44:10 ).
